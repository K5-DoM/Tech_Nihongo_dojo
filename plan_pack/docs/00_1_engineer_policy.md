# 00_1. エンジニア方針（Engineer Policy）

## 設計原則（3〜5個）

1. **契約駆動**  
   API・DB・Structured Output は docs（06_api_contracts, 05_db_schema, 07_prompt_design）を「契約」とし、実装は契約に従う。契約変更は意図とトレードオフを 00_2_decision_log に残す。

2. **先に価値検証、後で拡張**  
   「音声」「凝ったUI」より「評価品質」「継続率（UX）」を優先する。Week 単位の Gate 条件（セッション完了率・評価JSON失敗率・継続率）を満たすことを優先する。

3. **失敗を明示的に扱う**  
   JSON パース失敗・OpenAI タイムアウト・Webhook 失敗などは、リトライ・フォールバック・ユーザー向けメッセージを設計に含め、曖昧な握りつぶしをしない。

4. **説明可能性を残す**  
   技術選定・スキーマ・非機能の妥協点は理由付きで残す。就活面接で「なぜその判断をしたか」を説明できるレベルでドキュメントとコードコメントを書く。

5. **個人開発の制約を前提にする**  
   固定費・運用負荷・1人で回せる範囲を前提にし、過度な抽象化や将来の汎用性より「今のスコープで確実に動く」ことを優先する。

---

## 非機能要件の優先順位

| 優先度 | 項目 | 目安・判断 |
|--------|------|------------|
| 1 | **セキュリティ・監査性** | RLS 必須、Webhook 署名検証必須、会話・課金イベント追跡可能 |
| 2 | **評価品質・一貫性** | 5軸評価の安定出力、Structured Output 失敗率低減 |
| 3 | **体感応答・安定稼働** | 通常 5秒以内（LLM 除く）、稼働率 99% 目標（MVP） |
| 4 | **可読性・保守性** | 短期の実装速度より、変更理由が分かるコード・ドキュメント |
| 5 | **スケール・原価** | MVP 後（要約圧縮・非同期キュー・レート制限） |

---

## 変更時の判断ルール

- **可読性 > 短期速度**  
  一時的な早さのために意図が分からないコードを増やさない。リファクタは「理由とスコープ」を明記してから行う。

- **契約との整合**  
  新API・スキーマ変更は 06_api_contracts / 05_db_schema と整合させる。ずれが生じたら「意図的な拡張」か「設計ドキュメントの更新」のどちらかを明示する。

- **スコープ変更**  
  非目標（00_0）に手を出す場合は、意図・リスク・見直し条件を 00_2_decision_log に記録し、必要なら 03_dev_plan の優先度を調整する。

- **AI と人間の役割**  
  AI は実装・テストコード・ドキュメント案を出す。最終的な「採用するか・スコープを切るか」は人間が判断し、理由を残す。
