# 15. JTC向けポートフォリオ開発方針（4週間集中）

日本の大企業（JTC）技術職就活で「設計→実装→評価→運用」とLLM制御・認証/認可/セキュリティを説明できる完成度を、最短で最大化するための方針。  
**スコープ**: plan_pack 内の優先順位・ロードマップ・DoD・リスクの整理。コード変更は別フェーズ。

---

## 1. 結論（方針）

**方針: ピボット不要・方針調整**

### 理由

- **ドメイン・技術スタックはそのまま刺さる**  
  AI模擬面接・5軸評価・弱点履歴→次回反映は、JTCが求める「要件→設計→実装→評価→改善」の一気通貫を示せる。React/Vite/TS、Hono on Workers、Supabase（Auth/RLS）は説明しやすく、トレードオフも意思決定ログで語れる。

- **やるべきは優先度の付け直し**  
  現状の12週計画は「課金・成果物・β」まで含むが、就活で差をつけるのは「LLMの不安定性を契約・再試行・フォールバック・ログで制御していること」「認証/認可/RLS/運用を最低限きちんと入れていること」。  
  そのため **4週間（または20〜30時間）で「MVPの完成度」を上げ、課金・成果物生成は Could に回す** 方針が最短で価値が最大化する。

- **ピボットは不要**  
  サービスコンセプトを変える必要はなく、**何を Must にして何を後回しにするか** の調整だけで足りる。

## 2. 優先順位表（Must / Should / Could）

JTC向けに「面接で語れる」順で整理。Must を外さず、Should で完成度を足し、Could は時間があれば。

| 優先度 | 項目 | 内容 | 就活で刺さるポイント |
|--------|------|------|------------------------|
| **Must** | 認証・認可・RLS | 全API認証必須、Supabase RLSでユーザー行隔離 | 認証/認可の一貫設計とRLSの役割を説明できる |
| **Must** | Structured Output 実運用 | Zod契約でチャット/評価のJSONを保証し、**失敗率を計測できる状態**にする | LLM出力を契約で制御し、品質を数値で見ている |
| **Must** | リトライ・フォールバック | パース失敗時リトライ1回＋フォールバック文面、挙動をドキュメント化 | 失敗を握りつぶさず明示的に扱っている |
| **Must** | DoDの計測可能化 | 評価JSON失敗率・セッション完了率などを集計できる（ログ or 簡易メトリクス） | 「Done」を定義し、Gateで判断できる |
| **Must** | 最低限のCI | ビルド・リント・（手元）テストが通る状態をCIで担保 | 設計→実装の品質をCIで回している |
| **Must** | セキュリティの明文化 | API key管理・RLS・Webhook署名検証を 09_security と実装で一致させる | セキュリティを設計に含めている |
| **Should** | 監視・メトリクス | 4xx/5xx率・LLM呼び出し失敗率をログまたは簡易ダッシュボードで確認可能 | 運用を意識した観測可能性 |
| **Should** | 運用Runbook | 障害時・Webhook失敗時・手動対応の手順を 12_ops_runbook で具体化 | 運用まで考えた設計 |
| **Should** | ログ設計 | 構造化ログでLLM失敗・リトライ・フォールバック発動を追えるようにする | デバッグと改善の根拠を残す |
| **Should** | Gate A（α前） | セッション完了率≥70%、評価JSON失敗率≤3% を計測し通過確認 | 数値で品質を担保している |
| **Could** | Stripe課金 | Checkout・Webhook・チケット加算 | 課金は市場調査後に検討でよい |
| **Could** | 成果物生成API | 30秒自己紹介・60秒研究説明など | 差別化はMVP安定後に |
| **Could** | プロフィールUI | F1の画面実装 | 開始時は空で送れる前提で可 |
| **Could** | KPIダッシュボード | 原価・継続率の可視化 | 運用最適化フェーズで |

## 3. 4週間ロードマップ（週ごと成果物・タスク）

前提: 週5〜7.5h（合計20〜30時間）。過剰な機能追加はしない。

| 週 | 成果物 | タスク |
|----|--------|--------|
| **Week 1** | Structured Output 実運用開始 ＋ 失敗率「計測可能」 | ・チャット/評価のStructured Outputを実際に呼び出し、正常・失敗の両方を確認する ・パース失敗・リトライ・フォールバック発動をログに出す（またはカウンタで記録） ・DoDの「評価JSON失敗率」を集計する方法を決め、最低限のログまたはDB/メトリクスを用意する ・07_prompt_design の「パース失敗時1回リトライ＋フォールバック」が実装と一致しているか確認 |
| **Week 2** | リトライ・フォールバックの明文化 ＋ CI | ・リトライ・フォールバックの仕様を 06_api_contracts または 07_prompt_design に明記する ・実装がそれに沿っているか確認し、必要なら 00_2_decision_log に「握りつぶしをしない」方針を追記 ・CI（GitHub Actions 等）でビルド・リントを回す。テストは手元実行前提でも、CIでビルドが通ることを必須にする ・body パース失敗で 400 を返すなど、エラーハンドリングの一貫性を確認 |
| **Week 3** | 監視・メトリクス ＋ Runbook | ・4xx/5xx または LLM 失敗率を確認できるようにする（Workers のログ／外部ログ／簡易メトリクスいずれか） ・12_ops_runbook に「日次で見る指標」「LLM失敗が増えたときの確認手順」を追記 ・09_security_privacy と実装の対応（API key・RLS・Webhook署名）をチェックし、ずれがあればドキュメントを更新 ・RLSポリシー一覧（05_db_schema）と実際のポリシーが一致しているか確認 |
| **Week 4** | DoD確認 ＋ ドキュメント・α準備 | ・DoDチェックリスト（後述）をすべて満たしているか確認 ・Gate A（セッション完了率≥70%、評価JSON失敗率≤3%）を計測し、αテスト前の通過判定をする ・00_2_decision_log に「JTC向け4週間方針」を一行追記 ・面接で話す用の「デモ台本」または「設計→実装→評価→運用の説明順」を簡潔にメモ（README または plan_pack 内） ・必要なら小規模α（5人程度）の準備まで |

## 4. Done の定義（DoD チェックリスト）

以下を満たした時点で「JTC向けに説明可能な完成度」の Done とする。

- [ ] **Structured Output 失敗率が計測できる**  
  チャットまたは評価のJSONパース失敗（リトライ後も失敗した件数）を、ログ集計またはメトリクスで算出できる。

- [ ] **リトライ・フォールバックが実装され、挙動がドキュメントと一致している**  
  07_prompt_design / 06_api_contracts に「1回リトライ→失敗時はフォールバック」と書かれており、実装がそれに沿っている。

- [ ] **全APIが認証必須で、未認証で 401 になる**  
  requireAuth 等で /api/* が保護されていることを確認済み。

- [ ] **RLS が全対象テーブルで有効で、ポリシーが 05_db_schema と一致している**  
  profiles / interviews / messages / evaluations / weakness_history / billing_events の RLS と SELECT/INSERT/UPDATE 方針が説明できる。

- [ ] **CI でビルド・リントが通る**  
   main/develop への push または PR で、少なくともビルドとリントが成功する。

- [ ] **4xx/5xx または LLM 失敗が観測できる**  
  運用Runbookで「何を見て異常を判断するか」が書かれており、その情報がログまたはダッシュボードで取得可能。

- [ ] **セキュリティの前提が明文化されている**  
  API key は Workers Secret、Webhook は署名検証必須、RLS 必須が 09_security_privacy にあり、実装と対応している。

- [ ] **Gate A を計測できる**  
  セッション完了率・評価JSON失敗率を算出する方法が確定し、必要なら簡易スクリプトやクエリで集計できる。

## 5. リスクと対策（箇条書き）

### Workers 環境の制約

- **リスク**: CPU時間・メモリ・実行時間制限により、長時間LLM処理や大量DB書き込みでタイムアウトやメモリ不足になる可能性。
- **対策**: 長時間処理は「1リクエスト＝1ターン」に留め、評価生成（/finish）も1回の呼び出しで収める。トークン数・コンテキスト長を 07 で上限し、将来の要約圧縮は Could で検討。制約は 04_architecture または 00_2_decision_log に一言記載する。

### LLM JSON パース失敗

- **リスク**: Structured Output でも稀に不正JSONやスキーマ外フィールドが返り、パース失敗や Zod エラーになる。
- **対策**: 1回リトライ＋失敗時はフォールバック文面で応答を返す。**失敗・リトライ・フォールバック発動を必ずログに残し、失敗率を計測する**。失敗率が Gate（例: ≤3%）を超えたらプロンプト・モデル・温度の見直しを 07 に記載。

### RLS 設計の落とし穴

- **リスク**: API が Service Role で Supabase に接続していると RLS がバイパスされ、他ユーザーデータが見える。INSERT 時の `WITH CHECK` 漏れで意図しない行が入る。
- **対策**: API から Supabase へは **ユーザー JWT を渡して RLS を効かせる** 接続を使う。Service Role は管理用・Webhook 用など必要最小限にし、用途を 00_2_decision_log に明記。RLS ポリシーは 05_db_schema と実装を照合し、INSERT/UPDATE の `with check` を全テーブルで確認する。

### 認証漏れ

- **リスク**: 新規エンドポイント追加時に認証チェックを忘れ、未認証でデータ取得できる。
- **対策**: /api/* を一括で requireAuth するか、ルート登録を一箇所にまとめ「認証不要の例外」を明示する。00_2_decision_log の「全 /api/* を requireAuth で保護」を守る。

### ログ・メトリクスのコスト

- **リスク**: 無料枠で運用する場合、ログやメトリクスの保存量・問い合わせ量が枠を超える。
- **対策**: 最低限「LLM失敗回数」「リトライ回数」「フォールバック回数」をカウントする程度にし、サンプリングや集約で容量を抑える。詳細ログは障害調査時のみ有効化する運用を 12_ops_runbook に書く。

### 既存リスク登録との関係

- **R1（AI評価の誤判定）**: 根拠表示・再生成・免責は PRD/09 の通り。DoD では「失敗率の計測」で品質を可視化する。
- **R4（Webhook不整合）**: 署名検証・冪等処理は Must ではなく Could（課金を後ろにしたため）。課金を入れる時点で 13_risk_register の対策を実施する。
- **R6（開発遅延）**: 本ドキュメントの Must/Should/Could と 4 週ロードマップでスコープを固定し、Could は切る判断をしやすくする。

## 6. 既存 plan_pack との関係

- **03_dev_plan.md**: 12週間計画はそのまま参照用に残す。**4週間は「Phase 1 の完成度をJTC向けに最大化する集中期間」** として、本ドキュメントを優先する。
- **00_0_project_intent.md**: 目的（就活で語る価値・説明可能な開発経験）と整合。本ドキュメントはその「最短で完成度を上げる」具体化。
- **00_2_decision_log.md**: 方針変更や「4週間集中」「課金をCouldに」などの判断を、必要に応じて1行〜数行追記する。
- **13_risk_register.md**: 上記リスク・対策のうち、運用で常に持ち回るものは 13 に要約を追記してもよい。
